# 大模型及相关概念梳理

Large Language Model &amp; Ecosystem Concepts

## 大模型

**大模型（Large Language Model，LLM）**指参数规模达到数百亿的深度学习语言模型。通过对海量文本数据的预训练，模型能够学习语言统计规律、事实知识和一定的推理能力。

> **通俗理解：**大模型是一个读过全网几乎所有文字，学会"接话"的超级预测机。就像你手机输入法的联想功能，你打"今天天气真"，它猜"好"。只不过大模型这个"输入法"，是把整个互联网都"读"了一遍，你给它一段开头，它能接出一整篇有理有据的文章。它的本质就是在算下一个最可能的字。

当前具有代表性的模型包括 `GPT-4`、`Claude`、`DeepSeek`、`Qwen` 等。

## 智能体

**智能体（AI Agent）**是以大模型为核心计算引擎，具备环境感知、任务规划、工具调用和动作执行能力的自主程序系统。大模型输出文本，而智能体输出行动。

> **与大模型的核心区别：**智能体是让大模型具备行动能力的框架。它能理解目标、自主规划步骤、调用搜索引擎、代码解释器、数据库、第三方 API 等外部工具，并根据结果反思调整，直到完成任务。具备自我纠错和重试能力。

在记忆机制上，两者也有明显区别：大模型仅依赖当前会话的上下文窗口维持短期记忆。智能体除了利用上下文外，通常还集成**向量数据库**等长期记忆模块，可跨会话保存和检索信息。

> **比如：**如果要求"分析财报并生成 PPT"，大模型只能提供格式建议和文字分析，无法实际生成文件。智能体则可自主完成：读取文件 → 调用分析工具 → 整理关键发现 → 调用 PPT 生成接口 → 输出成品的工作流。

## 如何使用大模型

使用大模型的方式主要有三种：**网页对话、云服务 API 调用和本地部署**。不同的使用方式适用于不同的应用场景，各有优缺点。

### 1 网页对话

网页对话是最直接的方式。用户通过厂商提供的网页版聊天界面输入文本，模型实时返回结果。这种方式不需要任何技术基础，打开浏览器即可使用。

它的局限在于无法实现**自动化处理**，难以批量处理数据，功能也受到平台界面的限制。

### 2 云服务 API 调用

**应用程序接口（API）**调用是将大模型集成到自身系统的主要方式。通过 API，开发者可以在自己的程序里向模型服务端发送 HTTP 请求，传入提示词和相关参数，获取模型返回的结果并继续处理。这使得自动化、批量化、灵活的流程编排成为可能，模型能力可以被无缝嵌入到现有的业务系统或产品中。

为了进一步简化调用，厂商通常会提供特定编程语言的**软件开发工具包（SDK）**。SDK 对 API 进行了封装，把身份认证、连接管理、请求构造、流式响应处理、错误重试等细节包装成编程语言中的函数或对象。

> **关系梳理：**API 是通信的协议和接口，SDK 则是面向开发者的、基于该接口的工具包，两者相互配合以降低集成门槛。例如 OpenAI 提供的 Python SDK 允许开发者用几行代码就完成一次模型调用。

### 3 本地部署

本地部署是指将开源大模型的权重文件下载到自有服务器或本地计算机上运行推理。这种做法需要一定的硬件资源和运维能力，但在以下情况下是必要或更优的选择。

- **数据安全与隐私**  
  金融、医疗、政务等敏感数据行业，法规禁止数据外传。本地部署确保数据只在本地处理，完全不出域。
- **低延迟与成本控制**  
  高并发场景下云端 API 成本高且网络延迟不可控。有合适硬件时本地部署长期成本更低。
- **深度定制**  
  可直接微调模型、用私有数据优化表现，或修改推理代码改变行为，云端闭源 API 难以做到。
- **合规要求**  
  特定行业明确要求模型和数据处理必须在特定地理区域或受信环境内。

## RAG

**检索增强生成（RAG）**是一种将信息检索与文本生成相结合的技术架构。其工作流程为：用户提出问题后，系统先从外部知识库中检索出与问题最相关的文档片段，然后将这些片段与原始问题一同提交给大模型，由大模型阅读参考资料后生成最终回答。

**RAG 解决的核心问题：**

- Ⅰ **知识时效性不足**——大模型训练数据存在截止日期，RAG 可从实时更新的知识库中检索最新信息。
- Ⅱ **幻觉问题**——大模型在缺乏可靠依据时可能编造内容。RAG 将输出锚定在可追溯、可核查的来源上，显著降低幻觉概率。
- Ⅲ **私有知识问答**——企业内部文档无法参与公共模型预训练。RAG 允许在不微调模型的前提下查阅私有资料并生成精准答案。
- Ⅳ **专业领域覆盖不足**——小众专业问题知识储备不够，外部专业知识库可有效补充模型自身知识的缺口。

---

**嵌入模型（Embedding Model）**是 RAG 中执行检索功能的核心组件，本质上是参数较少的开源语言模型。嵌入模型能够将一段文本转换为一组高维向量，即文本向量化。语义相似的文本，其向量在向量空间中彼此靠近。语义差异较大的文本，其向量距离则较远。

> **RAG 检索过程：**系统预先用嵌入模型将所有待检索的文档转换为向量并存入向量数据库。当用户提问时，同一嵌入模型将问题也转换为向量，然后在向量数据库中通过相似度搜索找出与问题向量最接近的文档片段。被召回的相关片段会作为上下文提供给大模型，用于生成最终答案。

当前常用的嵌入模型包括 OpenAI 的 `text-embedding-3` 系列、开源的 `bge-large` 等。

## 提示词

在与大模型交互的开发实践中，提示词通常划分为两个层次：**系统提示词**和**用户提示词**。

- **系统提示词（System Prompt）**  
  即**人格提示词**，由开发者在应用层面预先设定，在整个对话会话或产品生命周期中保持不变。用于定义模型的基础角色、行为风格、回答边界和安全准则。确保角色行为的一致性和可控性。
- **用户提示词（User Prompt）**  
  即**对话提示词**，由最终用户在每次交互时动态输入。反映用户当前的具体需求或指令。模型在系统提示词的约束基础上，解析用户提示词的意图并生成回应。

> **设计理念：**这种分层设计将不变的"身份设定"与可变的"任务需求"分离。系统提示词保证应用的边界和行为一致性，用户提示词提供交互的灵活性和针对性，二者组合使模型既能维持稳定角色，又能应对多样化的实际任务。

## Skill

**Skill** 是对大模型使用外部工具、执行具体动作的能力所做的描述和实现。在实践中，Skill 的实现方式分为两类：**基于函数调用的 Skill**和**基于提示词的 Skill**。

### 基于函数调用的 Skill

这类 Skill 通过函数调用的机制实现。开发者将外部工具或函数的名称、功能描述和参数定义以结构化格式（通常为 JSON）提供给大模型。当模型判断用户意图需要调用某个工具时，它并不直接执行该工具，而是生成一个标准化的函数调用请求，由外部程序实际执行后将结果返回给模型，模型再基于返回结果继续生成回答。

> **举例：**查询实时天气或股价、操作数据库、发送通知、调用第三方 API 等。这类 Skill 需要配套的运行环境来承接函数调用的执行与结果回传，是实现智能体自主行动的基础。

### 基于提示词的 Skill

这类 Skill 不依赖外部工具调用，而是通过在系统提示词中详细描述任务的执行规则、步骤、输出格式和边界条件来赋予模型某种"技能"。模型在回答时严格遵循这些提示词中定义的规范来组织行为和输出。

> **举例：**可以定义一个"文本摘要 Skill"，提示词描述为："当用户要求进行摘要时，先提取核心事实，再按'背景-要点-结论'结构组织，字数不超过 200 字，使用中性客观的语气。"模型收到用户提供的原文后，按照这套规范直接生成摘要内容，整个过程仅依赖模型自身的生成能力，不涉及任何外部系统。
