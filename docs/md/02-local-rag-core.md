# 本地 RAG 核心搭建

Local RAG Core — Ingestion to Retrieval

在[上一章](./01-concepts.md)中，我们厘清了大模型、智能体、RAG、嵌入模型等核心概念。本章将把这些概念落地为实际运行的代码——从原始 Markdown 教学文档出发，经历嵌入模型部署、文档向量化和检索服务搭建，最终构建起一个可工作的本地 RAG 检索系统。

全程不依赖任何云端 API，所有组件均部署在本地机器上，数据完全不出域。

## 环境与依赖

在开始之前，需要准备好 Python 运行环境和必要的依赖包。

**Python 虚拟环境**用于隔离项目依赖，避免与系统 Python 或其他项目的包版本冲突。推荐使用 `venv` 模块创建：

```bash
python -m venv openai_env
source openai_env/Scripts/activate   # Windows（Git Bash）
```

进入虚拟环境后安装依赖：

```bash
pip install llama-index-core llama-index-readers-file \
            llama-index-embeddings-ollama llama-index-vector-stores-chroma \
            chromadb Flask
```

各依赖在 RAG 系统中的角色如下：

| 包名 | 角色 |
|------|------|
| `llama-index-core` | 文档加载、节点切分、索引管理——整个 RAG 流程的编排框架 |
| `llama-index-readers-file` | 文件目录读取器，递归扫描 `input/` 下的所有 `.md` 文件 |
| `llama-index-embeddings-ollama` | Ollama 嵌入模型的适配器，将文本请求转发给 Ollama 服务 |
| `llama-index-vector-stores-chroma` | Chroma 向量数据库的适配器，负责向量的存取 |
| `chromadb` | 轻量级向量数据库，本地持久化存储，无需独立服务端 |
| `Flask` | 检索服务的 HTTP 框架，对外提供 REST API 端点 |

## 文档数据：Markdown 知识源

**RAG 的质量天花板取决于知识库的质量。**无论检索算法多精妙、嵌入模型多先进，如果知识库本身内容稀疏、格式混乱或与问题领域不匹配，检索结果必然差强人意。因此，文档准备是 RAG 搭建中投入产出比最高的一步。

### 为什么用 Markdown

原始教学课件通常为 PDF 格式。PDF 适合阅读和打印，但作为 RAG 知识库的数据源存在两个痛点：一是文本提取可能丢失层级结构和格式信息，二是公式、表格在提取时容易变形。

**Markdown（`.md`）** 是更适合 RAG 的文档格式，原因有三：

- **层级清晰**：`#` `##` `###` 等标题标记天然携带文档的逻辑结构，切分工具可以直接按标题层级做语义切分，而非按固定长度暴力截断
- **纯文本友好**：无二进制编码，无需 OCR，嵌入模型能直接处理
- **可版本控制**：纯文本便于 Git 追踪变更、团队协作编辑

### PDF 转 Markdown

将 PDF 课件转为 Markdown 是数据准备的必要步骤。现代多模态大模型已经具备直接从 PDF 文件中读取文字、识别标题层级、转换公式的能力，可以胜任这项任务。实际操作中有两种方式：

| 方式 | 适用场景 | 操作 |
|------|---------|------|
| **网页端对话** | 少量文件、免费 | 将 PDF 上传至对话界面，提示模型"将此 PDF 转换为 Markdown 格式，保留原标题层级，公式用 LaTeX 书写"。大模型逐页读取并输出转换后的 Markdown，人工复核后保存。|
| **API 批量处理** | 文件较多、自动化 | 编写脚本调用大模型 API，传入 PDF 文件，指定输出格式为 Markdown。 |


### 文档内容与组织

本案的知识库覆盖电力电子技术课程的 **6 个核心模块**，每个模块为一个独立的 Markdown 文件：

## Ollama：本地嵌入模型运行环境

**Ollama** 是一个开源的大模型本地运行工具，它将模型的下载、量化、推理和 API 服务封装为一条命令，无需手动配置 GPU 驱动或深度学习框架。在 RAG 系统中，Ollama 充当嵌入模型的运行载体。

### 安装与启动

从 [ollama.com](https://ollama.com) 下载对应操作系统的安装包（支持 Windows / macOS / Linux）。安装完成后，Ollama 在后台启动服务，默认监听 `http://127.0.0.1:11434`。

使用以下命令验证安装：

```bash
ollama --version
```

### 拉取嵌入模型

嵌入模型的选择需要权衡**语义理解质量**和**推理速度**。参数规模大的模型精度更高，但在 CPU 上推理速度可能无法接受；参数过小的模型则可能无法准确捕捉中文专业术语的语义。

本案选用 **`qwen3-embedding:8b`**——阿里通义千问系列的嵌入模型，8B 参数规模，支持中英双语。在中文技术文档的语义检索任务上表现优秀，且能在主流消费级硬件上流畅运行。使用以下指令下载模型。

```bash
ollama pull qwen3-embedding:8b
```

拉取完成后，模型由 Ollama 自动管理存储、加载和推理。`qwen3-embedding:8b` 的输出为 4096 维向量——这个维度越高，向量能表达的语义细节越丰富，但存储和检索的开销也越大。4096 维是当前嵌入模型的主流水平，在精度和效率之间取得了较好的平衡。

使用以下命令查看当前已有模型：

```bash
ollama list
```

## 索引构建：切分 → 嵌入 → 入库

文档和嵌入模型都准备就绪后，下一步是将文档向量化并存入向量数据库——这个过程称为**构建索引**。本案的构建脚本 `src/build_index.py` 封装了完整流程，一次运行即可完成全部工作。

### 统一配置

所有可调参数集中在 `src/config.py` 中管理，构建脚本和检索服务共享同一份配置，避免参数不一致导致的问题：

```python
# 路径配置
DOC_DIR = "raw-docs/input"          # 文档源目录
PERSIST_DIR = "database/vector"     # 向量库持久化目录

# 向量库配置
COLLECTION_NAME = "smps_knowledge"  # 集合名称

# 嵌入模型
EMBED_MODEL = "qwen3-embedding:8b" # Ollama 模型名

# 检索默认参数
SIMILARITY_TOP_K = 5               # 默认返回 top-5

# 切分配置
MAX_NODE_LENGTH = 1200             # 超长节点二次切分阈值（字符）
CHUNK_SIZE = 800                   # 二次切分的片段大小
CHUNK_OVERLAP = 100                # 相邻片段的重叠长度

# 嵌入写入
BATCH_SIZE = 10                    # 每批嵌入条数
```

### 构建流程

执行构建命令：

```bash
python src/build_index.py
```

脚本按 5 个阶段依次执行，每个阶段都在终端输出进度和统计信息。

#### 阶段一：加载文档

使用 LlamaIndex 的 `SimpleDirectoryReader` 递归读取 `raw-docs/input/` 目录下的全部 Markdown 文件。每个文件被加载为一个 `Document` 对象，同时携带文件名等元数据。此阶段同时清理文档中残留的 `&emsp;` 全角空格缩进。

#### 阶段二上部：Markdown 层级切分 + 噪声过滤 + 路径注入

切分环节是整个流水线最关键的一步，它决定了后续检索的质量上限。本案采用**两级切分策略**，第一级为 Markdown 标题层级切分。

**切分：**使用 `MarkdownNodeParser` 按标题层级（`#` → `##` → `###` → `####`）将文档拆解为独立的文本节点。每个节点对应 Markdown 的一个逻辑段落——可能是一个章节的概述，也可能是某个子主题的详细展开。

> **举个例子：** `power-device.md` 中的 `## 全控型器件` 标题下有 `#### GTR 功率三极管` 和 `#### 功率 MOSFET` 两个子节点。`MarkdownNodeParser` 会将它们分别切分为独立的节点，而不是把"GTR"和"MOSFET"的内容混在一个大块里。这样，当用户问"GTR 的工作原理"时，检索能精准命中对应的节点，而不会被 MOSFET 的内容稀释。

**噪声过滤：**切分后的节点中会混入三类无意义内容——仅有标题无正文的空节点、正文为 `[TOC]` 的占位节点、YAML front matter 元数据节点。这些节点不含实质信息，须过滤移除，避免在检索时占据有效位置。

**路径注入：**在每个节点的正文前添加其在整个文档中的层级路径（如 `[定义 > 功率器件概述 > 功率器件的分类]`）。这一步骤让嵌入模型在向量化时感知到节点在文档结构中的位置——当用户问"功率器件有哪些类型"时，"定义/功率器件概述/功率器件的分类"这个路径信息帮助模型确认这段内容确实在回答分类问题。

#### 阶段二下部：超长节点二次切分

Markdown 标题切分可能产生过长的节点（例如包含大段介绍和公式的章节）。超过 `MAX_NODE_LENGTH` 的节点会触发**二次切分**：使用 `SentenceSplitter` 按句子边界进一步拆分，每段 800 字符，相邻段保留 100 字符的交叠区域。

> **为什么保留交叠区域？**如果严格按照 800 字符一刀切，可能会把一个完整的定义切成两半——上半段在片段 A 的末尾，下半段在片段 B 的开头。检索时只召回了片段 A，大模型看到的定义就不完整。100 字符的交叠相当于给每个片段两边各留一点"安全带"，保证不会有信息在切分边界处断裂丢失。

#### 阶段三：连接嵌入模型

初始化 `OllamaEmbedding`，发送一条测试文本验证连通性。若 Ollama 未启动或模型未拉取，构建在此处终止并给出明确错误提示。终端输出嵌入向量维度（4096），确认模型正常。

#### 阶段四：准备向量数据库

清理旧的同名集合（如存在），创建新的 Chroma 集合。相似度度量设置为**余弦相似度（Cosine Similarity）**——这是语义检索最常用的度量方式，它关注向量在空间中的"方向"而非"长度"：语义相近的文本其向量指向相近的方向，而与文本长度无关。

Chroma 内部使用 **HNSW（Hierarchical Navigable Small World）** 索引结构加速搜索。HNSW 是一种基于图的近似最近邻算法，在百万级向量规模下仍能保持毫秒级的检索速度。

#### 阶段五：逐批嵌入与写入

这是最耗时的阶段。将全部文本节点按每批 10 条送入 Ollama 嵌入模型，获取 4096 维向量后，连同原始文本和元数据（文件名、章节路径）一起写入 Chroma 向量库。终端实时显示进度和预估剩余时间：

```
进度: 230/437 (52%) | 耗时: 45s | 剩余: 40s
```

写入完成后，终端输出总耗时和向量库实际条目数，确认写入完整、数量一致。最终全部向量数据持久化在 `database/vector/` 目录中。

## 检索服务：Flask API

为了让其他应用（如聊天前端、自动化脚本）可以调用向量库的检索能力，我们通过 Flask 封装此服务为 HTTP API。服务启动后常驻内存，接收查询请求并返回检索结果。

### API 端点

服务默认监听 `http://127.0.0.1:5050`，可通过环境变量 `RAG_PORT` 指定端口：

```bash
python src/query_api.py              # 默认 5050
RAG_PORT=5055 python src/query_api.py  # 自定义 5055
```

| 方法 | 路径 | 说明 |
|------|------|------|
| `GET` | `/` | 服务说明页，显示向量库名称和条目数 |
| `GET` | `/health` | 健康检查，返回 `{"status":"ok"}` 及集合信息 |
| `POST` | `/retrieve` | 检索入口，请求体：`{"question":"...","top_k":5}` |

### 请求与响应

**请求示例**（`POST /retrieve`）：

```json
{
  "question": "什么是功率器件",
  "top_k": 3
}
```

**响应示例**：

```json
{
  "results": [
    {
      "file_name": "power-device.md",
      "section": "/功率器件概述/功率器件的分类",
      "score": 0.8765,
      "text": "..."
    }
  ],
  "context": "[参考资料 1] 文件: power-device.md  |  章节: /功率器件概述/功率器件的分类/\n[定义 > 功率器件概述 > 功率器件的分类]\n### 功率器件的分类\n功率器件通常工作于高电压、大电流条件下...\n\n[参考资料 2] ..."
}
```

- **`results`**：结构化检索结果数组，每项含 `file_name`（来源文件）、`section`（章节路径）、`score`（相似度，0~1）、`text`（完整节点文本），适合需要元数据的场景。
- **`context`**：按 `[参考资料 N] 文件: xxx | 章节: xxx` 格式拼接好的纯文本上下文，后跟完整节点正文，可直接注入大模型的系统提示词，作为回答的依据。

### 启动时的检查

服务启动时会依次执行两项检查：

- **Ollama 连通性检查**：向 `http://127.0.0.1:11434/api/tags` 发送请求，确认 Ollama 服务在线。若超时或拒绝连接，终止启动并提示用户先启动 Ollama。
- **向量库加载**：读取 `database/vector/` 目录下的 Chroma 持久化数据，建立索引对象。若目录不存在或数据损坏，同样终止启动并给出明确错误。
